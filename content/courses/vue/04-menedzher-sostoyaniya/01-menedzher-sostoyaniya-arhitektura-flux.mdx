# Менеджер состояния. Архитектура Flux

Менеджер состояния представляет собой сущность, хранящую состояние и бизнес-логику приложения независимо от дерева компонентов. По сути, это глобальное хранилище данных, доступное для чтения и записи из любой точки приложения — нечто вроде вечно доступного компонента.

Вам, вероятно, уже знакомы такие инструменты, как Vuex для Vue или Redux для React. Оба они построены на архитектурном паттерне Flux, который задаёт правила потока данных внутри приложения. Ниже приведена диаграмма, иллюстрирующая структуру и направление данных во Flux:

Flux состоит из четырёх частей, а данные движутся строго в одном направлении.

Если вы работали с Vuex, то помните, что менять состояние можно только через `mutations` — в терминологии Flux это `dispatchers`. Однако в Pinia разработчики решили отказаться от мутаций и оставили лишь `state` (состояние), `actions` (действия) и `getters` (геттеры для получения данных из хранилища).

Менеджер состояния предназначен для данных, к которым нужен доступ из любого компонента: информация о пользователе, данные, необходимые на нескольких страницах, или состояние сложных многоступенчатых форм. При этом данные, которые нужны только одному компоненту, лучше хранить локально, а не выносить в глобальное хранилище.

Из предыдущих разделов курса мы уже убедились, что передача данных через `props`/`emit` далеко не всегда удобна. При глубоко вложенных компонентах и общих данных между разными частями приложения контролировать поток информации становится всё сложнее, и приходится искать способы его упростить.

Рассмотрим альтернативные подходы, позволяющие обойтись без многоуровневого проброса `props`/`emit`.

## Composables

С помощью Composables из Composition API можно создавать глобальные переменные для хранения состояния. В SPA приложениях глобальное состояние можно расшарить следующим образом:

```javascript
import { reactive } from 'vue'

export const state = reactive({})
```

Тем не менее при серверном рендеринге такой подход может привести к [проблемам безопасности](https://vuejs.org/guide/scaling-up/ssr.html#cross-request-state-pollution).

## Provide/inject

Механизм `provide`/`inject` работает так: родительский компонент предоставляет данные через `provide`, а любой из его потомков может получить их через `inject`. Рассмотрим пример:

Parent.vue:

```xml
<template>
    <Child1 />
    <button @click="message = 'Новые данные'">Изменить</button>
</template>
<script setup>
  import Child1 from './Child1'
  import { provide, ref } from 'vue'

  const message = ref('Привет мир!')
  provide(/* ключ */ 'message', /* значение */ message)
</script>
```

Функция `provide` принимает два аргумента: ключ (строка или символ), по которому дочерние компоненты будут искать значение, и сами передаваемые данные. Один компонент может вызывать `provide` несколько раз с разными ключами, предоставляя различные значения.

Child1.vue:

```xml
<template>
    <Child2 />
</template>

<script setup>
import Child2 from './Child2'
</script>

```

Child2.vue:

```xml
<template>
  <div>
    {{ message }}
  </div>
</template>

<script setup>
import { inject } from 'vue'

const message = inject(/* ключ сообщения */'message')
</script>
```

Как видно из примеров, родительский компонент предоставил строку `message` через `provide`, а дочерний компонент с двойной вложенностью принял эти данные и отобразил их на экране.

# Менеджер состояния Pinia

Pinia — это библиотека-хранилище для Vue, позволяющая организовать совместный доступ к состоянию приложения из разных компонентов.

Ключевые преимущества Pinia:

1.  Интеграция с Vue Devtools:

*   отслеживание изменений состояния;
*   удобный просмотр данных хранилища в компонентах;
*   помощь при отладке приложения.

2.  Автоматическое обновление при разработке без перезагрузки страницы — Hot Module Replacement.
3.  Расширяемость за счёт плагинов.
4.  Полноценная поддержка TypeScript.
5.  Совместимость с серверным рендерингом (SSR).
