# Работа с HTTP

HTTP (Hypertext Transfer Protocol) -- это один из ключевых прикладных протоколов, предназначенных для передачи данных. Он лежит в основе обмена информацией в интернете: гипертекстовые документы содержат ссылки на другие ресурсы, и пользователь может свободно переходить между ними.

Само понятие «гипертекст» было предложено Тедом Нельсоном ещё в 1965 году в рамках [проекта Xanadu](https://ru.wikipedia.org/wiki/%D0%9F%D1%80%D0%BE%D0%B5%D0%BA%D1%82_Xanadu). Создание оригинальной спецификации HTTP приписывают Тиму Бернерсу-Ли и его коллегам из CERN -- Европейской организации по ядерным исследованиям.

## Методы запросов HTTP

В HTTP определён набор методов, каждый из которых описывает, какое действие нужно выполнить с ресурсом. Рассмотрим самые распространённые из них.

**GET** -- запрашивает представление ресурса. Запросы этого типа предназначены исключительно для извлечения данных.

**HEAD** -- работает аналогично **GET**, но сервер возвращает ответ без тела -- только заголовки.

**POST** -- передаёт сущность (например, пользователя, задачу, колонку) указанному ресурсу. Такой запрос нередко приводит к изменению состояния на сервере или вызывает побочные эффекты.

**PUT** -- полностью заменяет текущее представление целевого ресурса данными, переданными в теле запроса.

**PATCH** -- вносит в ресурс частичные изменения.

**DELETE** -- удаляет указанный ресурс.

## Работа с HTTP в рамках наших проектов

Ранее мы использовали моковые данные, загружая их из JSON-файлов. Теперь самое время перейти к полноценному взаимодействию с сервером по сети.

У нас уже развёрнут API-сервер на базе json-server. Локально он доступен по адресу `localhost:3000`. В файле `vite.config.js` настроен прокси, который перенаправляет запросы с фронтенда на API-сервер. По адресу [http://localhost:3000](http://localhost:3000) можно посмотреть доступные эндпоинты. json-server по умолчанию разрешает запросы с любых источников, поэтому проблем с CORS не возникнет.

Для авторизации мы будем применять JSON Web Token (JWT). При входе в систему сервер генерирует токен и отправляет его клиенту в ответе. JWT -- стандарт для создания и передачи подписанных и зашифрованных данных в формате JSON. Подпись формируется либо на основе секретной строки (вроде `koi594jfurUUHH`), либо с помощью пары открытого/закрытого ключа.

JWT состоит из трёх частей:

*   header -- определяет алгоритм, используемый для генерации подписи;
*   payload -- содержит передаваемые данные;
*   signature -- подпись для проверки целостности токена.

Нам потребуется реализовать авторизацию и [CRUD-операции](https://ru.wikipedia.org/wiki/CRUD) над различными сущностями (пользователи, статьи, комментарии и т.д.).

Среди современных инструментов для отправки HTTP-запросов выделяются два основных:

*   [Fetch](https://developer.mozilla.org/ru/docs/Web/API/Fetch_API) -- встроенный в браузер API, пришедший на смену `XMLHttpRequest` (XHR);
*   [Axios](https://github.com/axios/axios) -- популярный HTTP-клиент, построенный на `Promise`.

В проекте TaskBoard мы продемонстрируем работу с **fetch**, а во OnlinePizza -- с **axios**.

Определим ключевые требования к сетевому слою нашего приложения:

*   Независимость остального кода от конкретной реализации API-модуля, чтобы при необходимости можно было заменить axios на fetch без изменения логики Pinia и компонентов Vue.
*   Централизованное управление конфигурацией запросов.
*   Возможность задавать и менять базовый URL.
*   Гибкое управление заголовками (в том числе заголовком авторизации).
*   Единая обработка ошибок при выполнении запросов.
*   Удобное подключение API-модуля как в компонентах Vue, так и в хранилищах Pinia.
